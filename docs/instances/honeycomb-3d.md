# Honeycomb-3D Instance

**The 32-cell {4,3,5} BFS star — the first 3D holographic bridge instance, proving the bridge is dimension-agnostic.**

**Modules:** `Common/Honeycomb3DSpec.agda` (AUTO-GEN), `Boundary/Honeycomb3DCut.agda` (AUTO-GEN), `Bulk/Honeycomb3DChain.agda` (AUTO-GEN), `Bulk/Honeycomb3DCurvature.agda` (AUTO-GEN), `Bridge/Honeycomb3DObs.agda` (AUTO-GEN), `Bridge/Honeycomb3DEquiv.agda`

**Role:** The {4,3,5} BFS star is the repository's first **3D holographic bridge instance**. It proves that the generic bridge theorem (`Bridge/GenericBridge.agda`) is **dimension-agnostic**: the same Agda term that produces the enriched type equivalence for 2D pentagonal tilings ({5,4}) also works for 3D cubic honeycombs ({4,3,5}), with no modification and no awareness of the dimension change. The patch is grown by BFS from a central cube, producing a concentric shell of 32 cubes. All 26 cell-aligned boundary regions have min-cut 1 (trivial single-face cuts), so the instance demonstrates architectural compatibility with 3D geometry rather than deep holographic surfaces. Genuine multi-face Ryu–Takayanagi separating surfaces in 3D are first achieved by the Dense-50 patch ([`instances/dense-50.md`](dense-50.md)) using greedy max-connectivity growth.

## 2. Topology

The honeycomb-3d patch is the BFS neighborhood of a central cube in the {4,3,5} hyperbolic honeycomb. Each cube has 6 faces; in the {4,3,5} honeycomb, 5 cubes meet at every edge. The BFS growth adds all face-neighbors of the central cube (6 face-neighbors), then their face-neighbors, producing 32 cubes in a roughly spherical shell around the origin.

**The {4,3,5} Coxeter geometry:**

| Property | Value |
|----------|-------|
| Coxeter group | [4,3,5] with 4 generators |
| Gram matrix signature | (3,1) — compact hyperbolic |
| Cell stabilizer | [4,3] — order 48 (octahedral) |
| Face crossings per cell | 6 |
| Edge valence (full honeycomb) | 5 cubes per edge |
| Dihedral angle | π/2 = 90° |

The Coxeter group [4,3,5] acts on hyperbolic 3-space via reflections in the simple root basis. The cell stabilizer [4,3] (the octahedral group, order 48) preserves the central cube. The 6 face-crossing transformations are conjugates of the wall-crossing generator s₃ by elements of the cell stabilizer.

**Patch structure:**

| Count | Value |
|-------|-------|
| Cells (cubes) | 32 |
| Faces per cell | 6 |
| Internal shared faces | ~31 |
| Boundary faces | ~130 |

The BFS-shell topology produces a thin boundary: each boundary cube connects to the interior through approximately 1 shared face. This is why all min-cuts are 1 — there is no multiply-connected bulk geometry in a BFS-grown patch. The Dense growth strategy (Dense-50 and beyond) addresses this limitation by producing compact, densely-connected clusters where boundary regions require cutting through multiple internal faces.

## 3. Boundary Regions

The 26 cell-aligned boundary regions are **all singletons** — each is a single boundary cube. No multi-cell connected subsets were found by the region enumeration (at `max_region_cells=4`), because the BFS shell has sparse inter-boundary-cell connectivity: most boundary cubes are not adjacent to other boundary cubes within the patch.

**Region type:** `H3Region` with 26 constructors (`h3r0` through `h3r25`).

**Min-cut value distribution:**

| Min-cut value | Count |
|---------------|-------|
| S = 1 | 26 regions (all) |

All 26 regions have min-cut 1 because each singleton boundary cube connects to the interior through exactly one shared face. Severing that single face disconnects the boundary cube from its complement. This is the trivial case of the flow-graph min-cut — no genuine multi-face separating surface is involved.

**Comparison with the key 3D milestone:**

| Metric | Honeycomb-3D (BFS) | Dense-50 (Greedy) |
|--------|-------------------|-------------------|
| Cells | 32 | 50 |
| Growth strategy | BFS (concentric shell) | Dense (max-connectivity) |
| Min-cut range | 1 | 1–7 |
| Distinct min-cut values | 1 | 7 |
| Multiply-connected bulk | No | Yes |
| RT surface depth | Trivial (single-face) | Genuine (multi-face) |

The Dense strategy is the 3D analogue of "packing pentagons": local density drives holographic depth. The BFS strategy produces concentric shells where every boundary cell has approximately one internal face, giving trivial min-cuts. This architectural distinction was discovered by the multi-strategy comparison in `07_honeycomb_3d_multiStrategy.py`.

## 4. Min-Cut / Observable Agreement

All 26 pointwise agreement cases hold by `refl` because both `S-cut` and `L-min` return the same ℕ literal (1) for every region.

**Pointwise agreement:**

```agda
-- Bridge/Honeycomb3DObs.agda
h3-pointwise :
  (r : H3Region) →
  S-cut h3BdyView r ≡ L-min h3BulkView r
h3-pointwise h3r0  = refl
h3-pointwise h3r1  = refl
  ...
h3-pointwise h3r25 = refl
```

**Physical interpretation:** In 3D, the min-cut is a set of internal *faces* (a minimal separating surface), not edges. A min-cut value S = 1 means that severing a single shared cube-face disconnects the boundary region from its complement. This is the 3D discrete Ryu–Takayanagi correspondence in its simplest form: boundary entanglement entropy (1 unit) equals bulk minimal surface area (1 face).

The correspondence is trivially correct here but architecturally significant: the **same `GenericBridge` theorem** that proves S = L for the 2D star patch (10 regions, values 1-2) also proves S = L for this 3D patch (26 regions, all value 1). The bridge theorem never inspects whether the patch is 2D or 3D — it operates on the abstract `PatchData` interface.

## 5. 3D Edge Curvature

In 3 dimensions, the combinatorial curvature lives on **edges** (not vertices). At an interior edge e with valence v (number of cells meeting at the edge):

> κ(e) = 2π − v · (dihedral angle)

For the {4,3,5} honeycomb (dihedral angle π/2):

> κ(e) = 2π − 5 · (π/2) = −π/2

This negative edge deficit confirms the hyperbolic character of the honeycomb.

The `Bulk/Honeycomb3DCurvature.agda` module encodes edge curvature classes and verifies the total curvature sum by `refl`:

```agda
totalH3Curvature-check : totalH3Curvature ≡ ...
totalH3Curvature-check = refl
```

**Note on edge valences:** The Honeycomb3DCurvature module was generated by `06_generate_honeycomb_3d.py` using the Coxeter infrastructure from `05_honeycomb_3d_prototype.py`. The dihedral orbit enumeration in script 05 double-counted cell centers (each center appeared twice in the orbit), producing inflated in-patch valences (8, 10 instead of the correct 4, 5). This was identified and fixed in `07_honeycomb_3d_multiStrategy.py`, which introduced deduplicated orbit-center sets. All subsequent Dense patches (Dense-50 through Dense-1000) use the corrected infrastructure and report correct valences (all 12 central-cube edges at valence 5 for patches with sufficient cells).

The Honeycomb3DCurvature module **type-checks correctly** because the curvature values form a self-consistent ℤ arithmetic expression — the `refl` proof verifies that the class-weighted sum equals the expected total, regardless of whether the valences match the physical geometry. The curvature regression test serves its intended purpose (verifying ℤ arithmetic judgmental computation) even though the underlying valence data was computed from the pre-fix infrastructure.

### 6.4 Generic Bridge Integration

The honeycomb-3d patch is retroactively validated as an instantiation of `GenericEnriched` in `Bridge/GenericValidation.agda`:

```agda
h3PatchData : PatchData
h3PatchData .PatchData.RegionTy = H3Region
h3PatchData .PatchData.S∂       = S-cut h3BdyView
h3PatchData .PatchData.LB       = L-min h3BulkView
h3PatchData .PatchData.obs-path = h3-obs-path

h3-generic-witness : BridgeWitness
h3-generic-witness = GenericEnriched.abstract-bridge-witness h3PatchData
```

The generic bridge machinery produces the full enriched type equivalence (Univalence path, verified transport, `BridgeWitness`) automatically from the 26-region `PatchData`. The theorem never inspects whether the patch is 2D or 3D — it operates on the abstract `PatchData` interface with `RegionTy = H3Region`, `S∂ = S-cut h3BdyView`, `LB = L-min h3BulkView`, and `obs-path = h3-obs-path`.

This is the **first proof that the bridge theorem is dimension-agnostic**: the same ~30-line parameterized module that handles 2D pentagonal tilings handles 3D cubic honeycombs with zero modification.

## 10. What the Honeycomb-3D Patch Does NOT Exercise

- **Non-trivial min-cut range.** All min-cuts are 1 (trivial single-face cuts). Non-trivial ranges (1–7 and beyond) require the Dense growth strategy: Dense-50 ([`instances/dense-50.md`](dense-50.md)) achieves min-cut up to 7.

- **Orbit reduction.** The honeycomb-3d patch has only 26 regions — far below the threshold (~500) where orbit reduction is needed. Orbit reduction is first used by Dense-100 ([`instances/dense-100.md`](dense-100.md)) with 717 → 8 orbits.

- **`abstract` barrier.** The 26-case pointwise proof is small enough to normalize quickly without sealing. The `abstract` pattern is first used for the filled patch ([`instances/filled-patch.md`](filled-patch.md)) with 360 subadditivity cases.

- **Area law / Bekenstein–Hawking bound.** The discrete area law S ≤ area and the half-bound 2·S ≤ area are first verified on Dense-100 ([`instances/dense-100.md`](dense-100.md)) and Dense-200 ([`instances/dense-200.md`](dense-200.md)).

- **Multiply-connected bulk geometry.** The BFS-shell topology produces thin, non-entangled boundaries where every boundary cell connects to the interior through ~1 face. Multiply-connected bulk is first achieved by the Dense growth strategy.

- **Subadditivity / monotonicity structural properties.** These are exercised on the 6-tile star patch ([`instances/star-patch.md`](star-patch.md)) and the 11-tile filled patch ([`instances/filled-patch.md`](filled-patch.md)).

- **Parameterized step invariance / dynamics.** The dynamics layer operates on the 6-tile star with parameterized weights. See [`instances/star-patch.md`](star-patch.md).

- **Gauge connections / quantum superposition.** The gauge and quantum layers are exercised on the 6-tile star. See [`instances/star-patch.md`](star-patch.md).

- **Curvature-agnostic bridge (Wick rotation).** The Wick rotation is demonstrated between AdS ({5,4}) and dS ({5,3}) on the star patch. See [`instances/desitter-patch.md`](desitter-patch.md).

- **2D polygon-complex / vertex curvature.** The honeycomb-3d is 3D with edge curvature; the 2D vertex curvature and Gauss–Bonnet theorem are exercised by the filled patch ([`instances/filled-patch.md`](filled-patch.md)).

## 12. Historical Significance

The honeycomb-3d patch is the repository's **first 3D holographic bridge instance** — the proof that the formalization scales from 2D pentagonal tilings to 3D cubic honeycombs without any change to the proof architecture.

### 12.1 The Dimension-Agnosticism Discovery

The key insight from the feasibility prototype (`05_honeycomb_3d_prototype.py`) was that the flow-graph min-cut computation is **dimension-agnostic**: the same max-flow algorithm that computes S(A) on 2D border-edge cuts computes S(A) on 3D face-surface cuts. The `GenericBridge` theorem depends on exactly four inputs (a region type, two observable functions, a path between them) and never inspects the dimension of the underlying geometry.

This means the entire bridge infrastructure — `PatchData`, `GenericEnriched`, `orbit-bridge-witness`, `BridgeWitness` — works identically in 3D. The developer writes zero new proof code for the dimension change; only the Python oracle generates different data modules.

### 12.2 The BFS Limitation Discovery

The honeycomb-3d patch also revealed a critical limitation: **BFS-shell patches produce trivial holographic surfaces**. All 26 regions have min-cut 1, meaning the "minimal separating surface" is always a single shared face — not a genuine 2D surface cutting through the 3D bulk.

This led to the multi-strategy comparison in `07_honeycomb_3d_multiStrategy.py`, which tested four growth strategies (BFS, Dense, Geodesic, Hemisphere) and identified Dense (greedy max-connectivity) as the correct strategy for 3D holographic formalization:

| Strategy | Max min-cut at 50 cells | Density |
|----------|------------------------|---------|
| BFS-shell | 1 | 2.44 |
| **Dense** | **7** | **2.72** |
| Geodesic | 2 | 1.96 |
| Hemisphere | (saturates at 20 cells) | 3.00 |

The Dense strategy produces 7× deeper holographic surfaces than BFS at the same cell count. This finding launched the Dense patch series (Dense-50 → Dense-100 → Dense-200 → Dense-1000) that carries the resolution tower, area law, and Bekenstein–Hawking half-bound.

### 12.3 The Curvature Infrastructure Prototype

The honeycomb-3d curvature module (`Bulk/Honeycomb3DCurvature.agda`) was the first formalization of **3D edge curvature** in the repository. It established the encoding pattern (curvature in twentieths via ℚ₁₀ = ℤ, edge classes by in-patch valence, class-weighted sum verified by `refl`) that all subsequent 3D curvature modules follow.
