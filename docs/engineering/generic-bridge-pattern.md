# The Generic Bridge Pattern

**PatchData → BridgeWitness: One Proof, N Instances**

**Audience:** Proof engineers, Agda developers, and anyone implementing large-scale formal verification of parameterized combinatorial systems.

**Primary modules:** `Bridge/GenericBridge.agda` (`PatchData`, `OrbitReducedPatch`, `GenericEnriched`, `orbit-bridge-witness`), `Bridge/GenericValidation.agda`, `Bridge/BridgeWitness.agda`

**Prerequisites:** Familiarity with the holographic bridge construction ([`formal/03-holographic-bridge.md`](../formal/03-holographic-bridge.md)), the orbit reduction strategy ([`engineering/orbit-reduction.md`](orbit-reduction.md)), and the oracle pipeline ([`engineering/oracle-pipeline.md`](oracle-pipeline.md)).

---

## 1. The Pattern in One Sentence

> **If a proof about a concrete combinatorial structure depends only on an abstract interface, then the proof can be stated and verified once, and instantiated at every concrete size by an external oracle.**

This is the central proof engineering innovation of the repository. It separates the holographic correspondence into two independent concerns — **geometry** (handled by the Python oracle) and **proof** (handled by the generic theorem, proven once) — so that adding a new patch instance requires zero new hand-written Agda proof.

---

## 2. The Problem: Proof Duplication

Before the generic bridge pattern, each holographic patch required its own hand-written enriched equivalence module:

| Patch | Module | Proof Content |
|-------|--------|---------------|
| Star (6-tile) | `Bridge/EnrichedStarObs.agda` | ~120 lines: Iso, isoToEquiv, ua, uaβ, transport |
| Filled (11-tile) | `Bridge/FilledEquiv.agda` | ~80 lines: same pattern, different names |
| Honeycomb (3D) | `Bridge/Honeycomb3DEquiv.agda` | ~80 lines: same pattern, different names |
| Dense-50 | `Bridge/Dense50Equiv.agda` | ~80 lines: same pattern, different names |
| Dense-100 | `Bridge/Dense100Equiv.agda` | ~80 lines: same pattern, different names |

Every module followed the **exact same proof architecture**:

1. Define `S∂ LB : RegionTy → ℚ≥0` (specification-level observables)
2. Prove `obs-path : S∂ ≡ LB` (via `funExt` of pointwise `refl`)
3. Define `EnrichedBdy = Σ[ f ∈ (RegionTy → ℚ≥0) ] (f ≡ S∂)` and `EnrichedBulk = Σ[ f ∈ (RegionTy → ℚ≥0) ] (f ≡ LB)`
4. Build `Iso` from `obs-path` + `isSet (RegionTy → ℚ≥0)`
5. `isoToEquiv` → `ua` → `transport` → `uaβ`

Steps 3–5 are **identical** in every module. They depend on exactly **three inputs**: a region type, two observable functions, and a path between them. Nothing about pentagons, cubes, curvature, or tiling geometry appears. The five modules were copy-paste with search-and-replace on type names.

This duplication is not just inelegant — it is a **scaling barrier**. Each new patch requires writing a new Equiv module, even though the proof is structurally identical. For the {5,4} layer tower (6 levels at BFS depths 2–7), this would mean six more copy-paste modules.

---

## 3. The Solution: Factor Through an Abstract Interface

The generic bridge pattern factors the proof into:

1. **An abstract interface** (`PatchData`) capturing the minimum data needed for the bridge.
2. **A single parameterized proof** (`GenericEnriched`) that works for any `PatchData`.
3. **Per-instance data modules** (generated by the Python oracle) that instantiate the interface.

```
Per-instance (Python oracle):          Generic (proven ONCE):
┌──────────────────────────┐          ┌──────────────────────────┐
│  Common/*Spec.agda       │          │  Bridge/GenericBridge.agda│
│  Boundary/*Cut.agda      │ ───────▶ │                          │
│  Bulk/*Chain.agda        │  feeds   │  PatchData → BridgeWitness│
│  Bridge/*Obs.agda        │  into    │  (proven once, ~30 lines)│
└──────────────────────────┘          └──────────────────────────┘
                                                │
                                                ▼
                                      ┌──────────────────────────┐
                                      │  BridgeWitness           │
                                      │  (full enriched equiv    │
                                      │   + ua + transport)      │
                                      └──────────────────────────┘
```

The per-instance modules contain **only data** (region types, observable lookup tables, orbit classifications). The generic theorem handles **all proof** (Iso construction, round-trip homotopies, ua application, transport verification).

## 4. The Three Components

### 4.1 PatchData — The Abstract Interface

```agda
record PatchData : Type₁ where
  field
    RegionTy : Type₀
    S∂       : RegionTy → ℚ≥0
    LB       : RegionTy → ℚ≥0
    obs-path : S∂ ≡ LB
```

Four fields — a region type, two observable functions (boundary min-cut and bulk minimal chain), and a path between them. Nothing else. The region type may have 10 constructors (star), 90 (filled), 717 (Dense-100), 1246 (Dense-200), or 1885 ({5,4} depth-7) — the record is agnostic.

**Design decision:** The record lives in `Type₁` because it stores `RegionTy : Type₀` as a field. This is the correct universe level and matches the downstream `BridgeWitness` record.

### 4.2 GenericEnriched — The Parameterized Proof

```agda
module GenericEnriched (pd : PatchData) where
  open PatchData pd

  enriched-equiv     : EnrichedBdy ≃ EnrichedBulk
  enriched-ua-path   : EnrichedBdy ≡ EnrichedBulk
  enriched-transport : transport enriched-ua-path bdy-instance ≡ bulk-instance
  abstract-bridge-witness : BridgeWitness
```

This parameterized module proves the full enriched type equivalence — Univalence path, verified transport, `BridgeWitness` assembly — for **any** `PatchData`. The module is ~30 lines of Agda. It is written once, proven once, and never modified.

**Key architectural property:** The proof uses **contractibility of reversed singletons**. Both `EnrichedBdy = Σ[ f ] (f ≡ S∂)` and `EnrichedBulk = Σ[ f ] (f ≡ LB)` are contractible, so the transport step closes by `isContr→isProp` without inspecting the specific forward map. This is simpler and more generic than the explicit `isProp→PathP` construction used in the concrete bridge modules.

### 4.3 OrbitReducedPatch — The Oracle-Friendly Variant

```agda
record OrbitReducedPatch : Type₁ where
  field
    RegionTy  : Type₀
    OrbitTy   : Type₀
    classify  : RegionTy → OrbitTy
    S-rep     : OrbitTy → ℚ≥0
    L-rep     : OrbitTy → ℚ≥0
    rep-agree : (o : OrbitTy) → S-rep o ≡ L-rep o
```

For large patches (717+ regions), the observables are defined on a small orbit type and composed with a classification function. The `PatchData` is extracted automatically:

```agda
orbit-to-patch : OrbitReducedPatch → PatchData
orbit-to-patch orp = record
  { RegionTy = RegionTy
  ; S∂       = λ r → S-rep (classify r)
  ; LB       = λ r → L-rep (classify r)
  ; obs-path = funExt (λ r → rep-agree (classify r))
  }
```

The `obs-path` is assembled by the 1-line lifting: `funExt (λ r → rep-agree (classify r))`. See [`engineering/orbit-reduction.md`](orbit-reduction.md) for the detailed scaling analysis.

## 5. The One-Function Composition

The entire pipeline from oracle-generated data to proof-carrying `BridgeWitness` is a single function call:

```agda
orbit-bridge-witness : OrbitReducedPatch → BridgeWitness
orbit-bridge-witness orp =
  GenericEnriched.abstract-bridge-witness (orbit-to-patch orp)
```

This composes three steps:
1. `orbit-to-patch` — extracts `PatchData` from the orbit-reduced data
2. `GenericEnriched` — the parameterized module proving the enriched equivalence
3. `abstract-bridge-witness` — packages everything into a `BridgeWitness`

**No per-instance proof engineering is needed.** The Python oracle generates the `OrbitReducedPatch` data; `orbit-bridge-witness` produces the `BridgeWitness` automatically.

## 6. The Workflow: Adding a New Patch Instance

Adding a new holographic patch instance to the repository follows a fixed 4-step workflow:

### Step 1: Generate Data (Python Oracle)

Run the appropriate Python oracle script (or write a new one) to emit 4 Agda modules:

| Module | Content | Example |
|--------|---------|---------|
| `Common/*Spec.agda` | Region type, orbit type, classify function | `data D100Region : Type₀ where ...` |
| `Boundary/*Cut.agda` | `S-cut-rep` on orbits, `S-cut = S-cut-rep ∘ classify` | 8 clauses for Dense-100 |
| `Bulk/*Chain.agda` | `L-min-rep` on orbits, `L-min = L-min-rep ∘ classify` | 8 clauses for Dense-100 |
| `Bridge/*Obs.agda` | `pointwise-rep` (orbit-level refl), `pointwise` (1-line lifting) | 8 refl proofs |

**Total per-instance code:** ~50 lines of hand-written Agda (the Obs module) + the generated data modules.

### Step 2: Construct the OrbitReducedPatch (Agda)

In `Bridge/GenericValidation.agda` (or `Bridge/SchematicTower.agda`), assemble the orbit-reduced patch record from the generated modules:

```agda
d100OrbitPatch : OrbitReducedPatch
d100OrbitPatch .OrbitReducedPatch.RegionTy  = D100Region
d100OrbitPatch .OrbitReducedPatch.OrbitTy   = D100OrbitRep
d100OrbitPatch .OrbitReducedPatch.classify  = classify100
d100OrbitPatch .OrbitReducedPatch.S-rep     = S-cut-rep
d100OrbitPatch .OrbitReducedPatch.L-rep     = L-min-rep
d100OrbitPatch .OrbitReducedPatch.rep-agree = d100-pointwise-rep
```

This is a mechanical assembly — every field is a direct import from the generated modules.

### Step 3: Produce the BridgeWitness (Automatic)

A single function call produces the full proof-carrying bridge:

```agda
d100-generic-witness : BridgeWitness
d100-generic-witness = orbit-bridge-witness d100OrbitPatch
```

This line invokes the generic theorem, which proves the enriched equivalence, applies `ua`, verifies transport via `uaβ`, and packages everything into `BridgeWitness`. **Zero new hand-written proof.**

### Step 4: Register in the Tower (Optional)

If the new patch extends the resolution tower:

```agda
d100-tower-level : TowerLevel
d100-tower-level = mkTowerLevel d100OrbitPatch 8
-- The 8 is the max min-cut value (the "holographic depth")
```

The `mkTowerLevel` smart constructor calls `orbit-bridge-witness` internally, ensuring topological consistency: every tower level uses the same generic proof schema.

### Summary: Development Effort per Instance

| Component | Lines | Who |
|-----------|-------|-----|
| Python oracle script | ~200–500 | Developer (one-time per growth strategy) |
| Generated Agda data modules | ~500–2000 | Python (automatic) |
| OrbitReducedPatch assembly | ~10 | Developer (mechanical) |
| `orbit-bridge-witness` call | 1 | Developer (one line) |
| Tower registration | 1 | Developer (one line) |
| **New hand-written proof** | **0** | — |

## 7. Why It Works: Geometric Blindness

The generic bridge pattern works because the holographic correspondence depends on **strictly less structure** than the geometry of the underlying tiling.

The `GenericEnriched` module depends on exactly four inputs:

1. A type `RegionTy : Type₀`
2. A function `S∂ : RegionTy → ℚ≥0`
3. A function `LB : RegionTy → ℚ≥0`
4. A path `obs-path : S∂ ≡ LB`

It never inspects:

| What it never sees | Why it doesn't matter |
|---|---|
| Number of constructors in `RegionTy` | The proof is parametric in the type |
| Whether the patch is 2D or 3D | The min-cut algorithm is dimension-agnostic |
| Whether the curvature is positive or negative | The bridge operates on the flow graph, not on the embedding geometry |
| Whether bonds carry gauge-group representations | The bridge sees only scalar capacities |
| Whether the observable values come from lookup tables or orbit composition | Both produce the same `PatchData` |

This **geometric blindness** is not a weakness — it is a structural fact about the holographic correspondence. The bridge operates on the abstract flow graph (bond weights and min-cuts); the geometry enriches the topology without constraining it. This explains:

- **Why the Wick rotation works** ([`formal/08-wick-rotation.md`](../formal/08-wick-rotation.md)): the bridge is literally the same Agda term for both {5,4} (AdS) and {5,3} (dS).
- **Why the 3D extension works** ([`formal/04-discrete-geometry.md`](../formal/04-discrete-geometry.md)): the bridge is dimension-agnostic.
- **Why the gauge enrichment is orthogonal** ([`formal/05-gauge-theory.md`](../formal/05-gauge-theory.md)): the bridge sees only `dim(ρ) : ℕ`, not the gauge group.
- **Why the quantum lift is trivial** ([`formal/07-quantum-superposition.md`](../formal/07-quantum-superposition.md)): the bridge is consumed by `cong₂` in a 5-line list induction.

## 8. Comparison with Alternative Approaches

### 8.1 Copy-Paste Per Instance (What We Replaced)

The pre-generic approach: copy the enriched equivalence module, search-and-replace type names, re-prove the Iso and transport.

| Property | Copy-Paste | Generic Bridge |
|---|---|---|
| Per-instance proof effort | ~80 lines of Iso/Equiv/ua | 0 lines (one function call) |
| Risk of copy-paste errors | High | Zero (parameterized module) |
| Scaling to 12+ instances | O(N) hand-written modules | O(1) generic module + O(N) data |
| Consistency guarantee | Manual (hope all copies agree) | Structural (same proof term) |
| Retroactive validation | Manual (check each copy) | `GenericValidation.agda` (type-checks 6 at once) |

### 8.2 Reflection / Metaprogramming

Agda supports term-level metaprogramming via the `TC` monad. One could imagine a tactic that inspects the region type and automatically generates the Iso.

| Property | Reflection | Generic Bridge |
|---|---|---|
| Trusted code | Reflection API + tactic code | Just the generic module (~30 lines) |
| Debugging | Opaque generated terms | Transparent parameterized proof |
| Performance | Reflection overhead at type-check time | Zero overhead (direct instantiation) |
| Library dependency | `Cubical.Reflection.*` (unstable API) | `Cubical.Foundations.*` (stable) |

The generic bridge avoids reflection entirely. The proof is a standard parameterized Agda module — readable, debuggable, and stable across library versions.

### 8.3 Inductive Tiling Definition

The original plan (§5.2 of the historical development docs) proposed defining an inductive type `HaPPYPatch (n : ℕ)` that recursively generates the graph and simplicial complex for an n-layer {5,4} patch. The bridge would then be proven by structural induction on the tiling.

This approach collides with a fundamental obstruction: the {5,4} tiling's closed-loop geometry (gap-filler tiles wrapping around at each layer) requires weeks of graph-theory formalization before any min-cut reasoning can begin.

The generic bridge **changes the structure of the induction**:

> **Original:** Structural induction on the **geometry** (layer-by-layer polygon complex).
>
> **New:** Structural induction on the **proof schema** (generic theorem applied to oracle-generated instances).

The hard geometry stays in Python. The Agda proof is written once. The "induction" is in the oracle's growth function, and the "induction step" is the generic theorem applied to the next level's data.

## 9. The BridgeWitness Record

All bridge results are packaged into a single universal record:

```agda
record BridgeWitness : Type₁ where
  field
    BdyTy              : Type₀
    BulkTy             : Type₀
    bdy-data           : BdyTy
    bulk-data          : BulkTy
    bridge             : BdyTy ≃ BulkTy
    transport-verified : transport (ua bridge) bdy-data ≡ bulk-data
```

This record is the **Milestones 3–4 deliverable**: a machine-checked proof that boundary and bulk observable packages are exactly equivalent types, with verified computable transport. It is defined in its own leaf module (`Bridge/BridgeWitness.agda`) to break the dependency chain: generic infrastructure depends only on abstract interfaces, not on per-instance implementations.

The record is consumed by:
- `Bridge/SchematicTower.agda` — tower assembly (`TowerLevel.bridge`)
- `Bridge/WickRotation.agda` — dS/AdS coherence (`shared-bridge`)
- `Bridge/GenericValidation.agda` — retroactive consistency (6 instances)
- `Causal/CausalDiamond.agda` — causal diamond endpoints carry tower levels with bridges

## 11. Integration with Other Engineering Patterns

### 11.1 The Oracle Pipeline

The generic bridge pattern integrates directly with the Python oracle pipeline ([`engineering/oracle-pipeline.md`](oracle-pipeline.md)). Each oracle script generates the data modules that feed into `OrbitReducedPatch`; the generic theorem handles the proof. This is the same division of labor used by the Four Color Theorem (Coq) and the Kepler Conjecture (HOL Light): external computation finds proofs, a simple kernel checks them.

### 11.2 The `abstract` Barrier

The generic bridge pattern is **complementary** to the `abstract` barrier ([`engineering/abstract-barrier.md`](abstract-barrier.md)):

- **Generic bridge** handles the RT correspondence `S = L`: proof cases = orbit count (very small: 2–9).
- **`abstract` barrier** handles inequality bounds where values differ across regions: proof cases = region count (large: 360–1246), but sealed to prevent downstream normalization.

Both patterns operate on the same patches. For Dense-100:

| Proof | Technique | Cases |
|-------|-----------|-------|
| S = L (RT correspondence) | Generic bridge via orbit reduction | 8 orbit `refl` proofs + 1-line lifting |
| S ≤ area (area law) | `abstract` barrier | 717 `(k , refl)` proofs, sealed |
| 2·S ≤ area (half-bound) | `abstract` barrier | 717 `(k , refl)` proofs, sealed |

### 11.3 The SchematicTower

The generic bridge feeds directly into the `SchematicTower` infrastructure ([`formal/11-generic-bridge.md`](../formal/11-generic-bridge.md) §5). Each tower level is constructed by `mkTowerLevel`, which calls `orbit-bridge-witness` internally:

```agda
mkTowerLevel : OrbitReducedPatch → ℚ≥0 → TowerLevel
mkTowerLevel orp mc = record
  { patch  = orp ; maxCut = mc
  ; bridge = orbit-bridge-witness orp }
```

The tower carries fully proof-carrying `BridgeWitness` records at each level. Adding a new level to the tower is a one-line call to `mkTowerLevel` — no new proof.

## 12. The Pattern as a Design Principle

The generic bridge pattern embodies a design principle that extends beyond holographic duality:

> **If a proof about a concrete combinatorial structure depends only on an abstract interface, then the proof can be stated and verified once, and instantiated at every concrete size by an external oracle.**

This pattern is well-known in software engineering (interfaces and dependency injection) but underexplored in formal verification of mathematical physics. Key properties:

1. **Separation of concerns.** Geometry (which varies per instance) is cleanly separated from proof (which is universal). The interface (`PatchData`) is the contract between the two.

2. **External oracle + simple kernel.** The Python oracle handles all combinatorial case enumeration (Coxeter reflections, BFS growth, max-flow computation, orbit classification). Agda checks the emitted `(k, refl)` witnesses individually. The oracle is the search engine; Agda is the checker.

3. **Logarithmic proof scaling.** Adding tiles to a patch only grows the `classify` function (one clause per region constructor), not the proof obligations (one `refl` per orbit representative). The orbit count grows logarithmically with the min-cut range.

4. **Retroactive validation.** The pattern can be applied retroactively to existing bridges. `Bridge/GenericValidation.agda` validates all six pre-existing bridge instances as instantiations of `GenericEnriched` — confirming that the generic architecture subsumes all existing bridges without modifying any of them.

5. **Composability.** The `BridgeWitness` record is consumed by multiple downstream systems (towers, Wick rotation, causal diamonds) without any of them knowing how the witness was produced. The generic bridge is an implementation detail hidden behind the abstract interface.
